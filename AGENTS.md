udplbd2 is the control plane for the EJFAT data plane, which is called udplb. It provides a gRPC API ([proto definition](proto/loadbalancer/loadbalancer.proto), [handlers](src/api)) for managing UDP-based load balancers that forward all traffic for a given "tick" to the same host (dataplane [protocol](src/dataplane/protocol.rs)). It accomplishes this by controlling a P4 program called `udplb` developed using the ESnet SmartNIC project. `udplbd2` updates the [P4 tables](src/snp4/rules.rs) using the [smartnic-p4 gRPC interface](src/snp4/client.rs) by generating a valid set of rules from the [current database state](src/reservation/active_reservation.rs). In order to update the load balance distribution for future ticks without disrupting the current calendar, udplbd2 receives [synchronization messages](src/reservation/event_id_sync.rs) from the sender.

udplbd2 includes extensive tooling to assist with observability, testing, development, including a [frontend](frontend) ([REST API](src/api/rest.rs)), CLI tools ([dataplane](src/dataplane/cli.rs), [API](src/api/cli.rs), [sncfg](src/sncfg/cli.rs), [main](src/main.rs)), [Prometheus metrics](src/metrics.rs), a [mock implementation](src/dataplane/mock.rs) of udplb that exposes a mock smartnic-p4 interface.

Code comments should only be written for documentation purposes and for confusing areas. Documentation quality should be high. Any changes to config options should be documented in both the [example config](etc/example-config.yml) and [configuration docs](docs/src/configuration.md). New Prometheus metrics should be documented in the [observability docs](docs/src/observability.md). Code should be tested, if adding new major features, be sure to add corresponding [doctor integration tests](src/dataplane/doctor.rs), CLI commands, frontend interface, and documentation.

Whenever you test, use `-r`. Use `cargo install --path .` rather than `cargo build` so that you can use the `udplbd` binary for your testing.
